# Поиск статей справки Авито по запросу пользователя

Решение этапа retrieval для RAG-пайплайна поддержки: по тексту вопроса
пользователя вернуть ранжированный топ-10 `article_id` из базы статей справки.

**Метрика:** MAP@10 на 500 калибровочных запросах.

## Структура

```
avito_rag/
├── candidate_data/          # исходные данные (articles.f, calibration.f, test.f)
├── src/
│   ├── common.py            # чистка HTML, лемматизация, метрика MAP@10
│   └── baseline_bm25.py     # локальный baseline: BM25 → MAP@10 = 0.270
├── kaggle_kernel/
│   ├── avito_search.py      # основное решение (запускалось на Kaggle GPU)
│   └── kernel-metadata.json
└── answer.csv               # итоговые предсказания для test.f
```

## Подход

### 1. Обработка HTML

Статьи хранятся с разметкой (таблицы, вкладки, спойлеры). Используется
`BeautifulSoup`: удаляются `script`/`style`, извлекается плоский текст
(`get_text`), схлопываются пробелы и `&nbsp;`. Служебные элементы (вкладки,
спойлеры) при этом превращаются в обычный текст — их содержимое полезно для
поиска, поэтому оно не выбрасывается.

Длинные статьи (до 900 тыс. символов) режутся на чанки по 1200 символов с
перекрытием 200; к каждому чанку добавляется заголовок статьи. Скор статьи по
dense-модели — максимум по её чанкам.

### 2. Источники релевантности (5 компонент)

| Компонента | Описание | MAP@10 отдельно |
|---|---|---|
| `bm25` | BM25Okapi по леммам (pymorphy3), заголовок с бустом ×3 | 0.270 |
| `char` | TF-IDF на символьных n-граммах (3–5, `char_wb`) — устойчив к опечаткам («возрат») | 0.352 |
| `dense` | `intfloat/multilingual-e5-large` (560M параметров < 1B), косинус запрос↔чанк, max по чанкам | 0.376 |
| `knn` | k=20 ближайших калибровочных запросов по e5-эмбеддингам, голоса за их ground truth с весом sim³ (leave-one-out при оценке) | **0.614** |
| `prior` | частота статьи в ground truth калибровки (в разметке всего 79 уникальных статей из 793 — запросы сильно сконцентрированы) | 0.318 |

**Итог смешивания (calibration, LOO для knn): MAP@10 = 0.679.**
Подобранные веса: `bm25=0.7, char=0.2, dense=1.0, knn=0.5, prior=0.1`.
(точные значения печатаются кернелом — см. `report.txt` в выводе)

### 3. Смешивание и подбор весов

Каждая матрица скоров нормируется в [0,1] по запросу. Веса компонент
подбираются **координатным подъёмом по MAP@10 на calibration**. Чтобы не было
утечки, компонента `knn` на калибровке считается в **leave-one-out** режиме
(сам запрос исключается из соседей).

### 4. Реранкинг

Топ-40 кандидатов stage-1 реранжируются кросс-энкодером
`BAAI/bge-reranker-v2-m3` (568M параметров < 1B): вход — запрос + заголовок и
первые 1500 символов статьи. Финальный скор: `norm(stage1) + α · sigmoid(rerank)`,
α подбирается на калибровке.

На этих данных перебор дал **α = 0**: чисто текстовый реранкер не улучшает
смесь, потому что сигнал `knn`/`prior` (какие статьи реально запрашивают под
похожие вопросы) сильнее лексического сходства «вопрос↔тело статьи». Реранкер
оставлен в пайплайне как страховка — на новых данных без калибровочного
покрытия он может помочь; вес выбирается автоматически.

### 5. Ограничения задачи

- Обе модели open-source, < 1B параметров, скачиваются с HuggingFace и
  выполняются локально в среде Kaggle — запросы пользователей во внешние API
  не отправляются.
- Ручной разметки теста и хардкода query_id нет: калибровка используется
  только как обучающие данные (kNN-голосование, prior, подбор весов).

## Анализ ошибок

- Чистый BM25 (0.27) проседает на разговорных формулировках и опечатках —
  добавлены char n-граммы и dense-модель.
- Запросы сильно сконцентрированы на популярных темах (79 статей покрывают всю
  калибровку) — prior и kNN по размеченным запросам дают большой прирост.
- Длинные статьи «топили» релевантные короткие в BM25 — dense-скор по чанкам
  с max-агрегацией это компенсирует.

## Воспроизведение

Kaggle (GPU T4, internet on), данные подключены как датасет с тремя `.f` файлами:

```bash
python kaggle_kernel/avito_search.py   # пути в скрипте: /kaggle/input/avito-support-search/
```

Сиды зафиксированы (`SEED=42`), результат детерминирован; скрипт печатает
MAP@10 каждой компоненты, подобранные веса и сохраняет `answer.csv` и `report.txt`.

Локальный baseline: `python src/baseline_bm25.py`.
