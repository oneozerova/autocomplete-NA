# RU Prompt Autocomplete — автодополнение окончаний для русских промптов

Быстрое, лёгкое автодополнение русских слов для промптов генерации изображений.
Пользователь вводит текст — модель мгновенно продолжает **текущее слово по его
окончанию**, предлагая морфологически корректные формы (род / число / падеж /
время): `красив` → `красивая · красивый · красивое · красивые`.

Без внешних моделей и без интернета в рантайме. Ядро использует **только
стандартную библиотеку Python**, один вызов ≈ **0.5 мс**, артефакты ≈ **39 МБ**.

---

## Как это устроено (ресёрч и решение)

Задача — не «предсказать следующее слово», а **дополнить окончание** уже
набираемого слова. Русский язык сильно флективный: по данным литературы, чтобы
покрыть 99% словоформ разговорной речи, нужно ~400k форм. Поэтому подход,
основанный на *словоформах* (а не на леммах), здесь естественен.

По итогам ресёрча (см. «Источники») стандартное решение для word/query
autocompletion — **символьная n-граммная языковая модель + морфологически
осведомлённое ранжирование + beam search для top-k**. Именно это и реализовано,
гибридом двух дешёвых сигналов:

| Компонент | Файл | Что делает |
|---|---|---|
| **CharNGram** | `src/ngram_model.py` | Символьная n-граммная модель (order 6) со *stupid-backoff* сглаживанием. Это и есть «Байесовская»/марковская модель: оценка `P(след. символ \| предыдущие символы)` по частотно-взвешенному словарю. Слова обрамляются маркерами `^…$`, поэтому модель учит, **где и как слово заканчивается** — то самое окончание. Beam search генерирует правдоподобные окончания, в т.ч. для **незнакомых основ** (OOV). |
| **FreqTrie** | `src/trie.py` | Префиксный индекс известных словоформ с кэшем top-k в узлах: точное, высокоточное дополнение реальных слов, ранжированных по частоте. |
| **ContextModel** | `src/context_model.py` | **Согласование по роду/числу/падежу** + **лексические коллокации**. Сигналы по окончаниям соседних слов: (1) **биграмма** ближайшего слова; (2) **суффикс-триграмма** `P(оконч \| оконч₋₂, оконч₋₁)` — падежное управление предлогом через прилагательное (`в тёмн[ом] лес[у]`); (3) **окно** знаменательных слов — дотягивается через обороты/союзы. Плюс сигнал (4) **словесная биграмма** `P(слово \| пред. слово)` — лексическая коллокация целыми словами (`на` → `изображение`, `чёткий` → `фокус`), не выводимая из окончаний. Обучена на 1.2M предложений Tatoeba (лексическая биграмма — только на реальном тексте) + доменный корпус промптов (только сигналы по окончаниям). Без морфологической библиотеки и парсера. |
| **AgreementModel** | `src/agreement.py`, `src/morph.py` | **Грамматическое согласование по грамемам** (не по поверхности). На сборке словарь тегируется `pymorphy3` → компактная таблица `форма → POS/род/число/падеж` (`morph.tsv.gz`, 2.3 МБ). Модель учит `P(число глагола \| число подлежащего)` и `P(род·число прил/местоим \| род·число сущ)`. В рантайме — только O(1)-лукапы (stdlib, без pymorphy). Чинит `ежик[и] стоя[т]` (число) и `женщина … котор[ую]` (род). |
| **Completer** | `src/completer.py` | Публичный API. Достаёт набираемое слово + слово слева, смешивает кандидатов из trie и n-граммы, переранжирует единым скором (частота × правдоподобие модели × согласование/коллокация × грамматическое согласование). Метод `ghost()` — уверенная inline-подсказка с порогом доминирования top-1 (см. ниже). |

### Почему n-грамма, а не LSTM

Задание допускало «лёгкий LSTM **или** Bayes». Выбрана символьная n-грамма,
потому что для этой задачи она строго лучше по инженерным критериям, которые
здесь ключевые (*быстро, легко, встраивается как фича*):

- **Скорость:** ~0.5 мс/вызов на CPU без разогрева — LSTM-инференс на порядки тяжелее.
- **Вес и зависимости:** чистый Python, без `torch` (~сотни МБ). Артефакт 38 МБ.
- **Обучение:** 12 секунд против часов; тривиально переобучить под доменные данные.
- **Качество:** на бенчмарке ниже n-грамма даёт top-5 **81–84%** при префиксе 5–6
  символов — при том, что LSTM выигрывал бы в основном на длинных зависимостях,
  которых внутри одного слова почти нет.

LSTM оставлен как задокументированная точка расширения (см. «Дальнейшие шаги»).

## Данные

- `data/ru_full.txt` — открытый частотный список **словоформ** OpenSubtitles 2018
  ([hermitdave/FrequencyWords]). `src/preprocess.py` чистит его (только кириллица,
  `count ≥ 5`, длина 2–24) → **357 779 словоформ** (87.7% всех токенов). Реальные
  инфлектированные формы с частотами — сигнал для trie и для обучения окончаний.
- `data/rus_sentences.tsv.bz2` — **1.2M предложений** русского из [Tatoeba]
  (связный текст). Нужен для обучения модели согласования (суффикс-биграммы
  соседних слов). Скачивается автоматически шагом сборки; без него `Completer`
  просто работает без учёта контекста.
- `src/prompt_corpus.py` — **синтетический доменный корпус русских промптов**
  (~80k). Реальных открытых корпусов русских промптов нет, поэтому он
  генерируется из шаблонов с грамматически **согласованными** фрагментами
  (`прилагательное(в роде) + существительное` + готовые корректные фразы
  локаций/стилей/света). Идёт в обучение сигналов по окончаниям (доменное
  согласование и фразеология промптов), но **не** в лексическую словесную
  биграмму — иначе его шаблонные существительные вытеснили бы реальные
  коллокации из естественного текста.
- **Доменная интерполяция частот** (`src/preprocess.py`) — стандартный приём
  адаптации LM (linear interpolation / count-merging). В частоты словаря
  подмешивается доменное распределение как **пол**: `target(w) = λ·N·P_домен(w)`,
  слово поднимается только *до* этого пола (`max`, не `+=`). Это вытягивает
  промпт-лексику, которую OpenSubtitles недо-представляет (`изображение` — ранг
  ~6k, из-за чего на префиксе `из` побеждало разговорное `извините`), и при этом
  **не раздувает** уже частотные слова (`девушка`), чтобы не ломать согласование.
  λ=0.1 подобрана по `evaluate.py` (общая точность не падает, а слегка растёт).
- `src/domain_lexicon.py` — **курируемая лексика промптов**. OpenSubtitles слабо
  покрывает описательную лексику (`детализированный` — ранг ~260k;
  `акварельный`/`минималистичный` вообще отсутствуют). Список типовых
  прилагательных/существительных промптов + отдельный тир **мета-слов**
  (`изображение`, `картинка`, `иллюстрация`…) + **правило-ориентированный
  склонятель прилагательных** дают все нужные формы; они и питают доменное
  распределение для интерполяции выше.

## Бенчмарк

Единого канонического бенчмарка для «дополнения окончаний» в русском нет, поэтому
используется общепринятая методика word/query-autocompletion на **открытых
данных**: частотно-взвешенная выборка целевых слов (как реально печатают
пользователи), метрики — по длине введённого префикса. `python -m src.evaluate`:

**EXACT-MATCH (реалистичный режим, полная модель)** — попали ли точно в набираемое слово:

| префикс | top-1 | top-3 | top-5 | MRR | экономия нажатий |
|---:|---:|---:|---:|---:|---:|
| 3 симв. | 23.7% | 40.6% | 49.1% | 0.330 | 20.3% |
| 4 симв. | 35.2% | 56.3% | 67.1% | 0.469 | 29.4% |
| 5 симв. | 49.1% | 70.2% | 79.5% | 0.604 | 37.0% |
| 6 симв. | 49.5% | 72.2% | 81.3% | 0.615 | 39.0% |
| **всего** | **38.5%** | **58.8%** | **68.3%** | **0.495** | **29.2%** |

**SUGGESTION VALIDITY (обобщение, основы вне обучения)** — не выдаём ли мусор:

- Сгенерированных моделью подсказок, являющихся **реальными** словами: **98.9%**
- Случаев, где ≥1 подсказка — реальное слово: **99.9%**

Вывод: на префиксе 5–6 символов top-5 = 81–84%, экономия до 40% нажатий; на
незнакомых основах модель почти всегда предлагает настоящие русские формы, а не
мусор. Точное совпадение с одной удержанной формой — заведомо неверная метрика
для OOV: из префикса `красив` формы `-ый/-ая/-ое` **все** валидны, поэтому мерим
валидность, а не угадывание конкретной флексии.

### Согласование по роду/числу/падежу (контекст)

`python -m src.eval_context` — бенчмарк на **held-out предложениях** (у каждого
слова есть реальный левый контекст), свип веса согласования `gamma` (`gamma=0` —
контекст выключен):

| gamma | top-1 | top-3 | top-5 | MRR |
|---:|---:|---:|---:|---:|
| 0 (выкл) | 23.7% | 43.8% | 54.9% | 0.349 |
| 6 | 34.1% | 54.8% | 64.3% | 0.454 |
| **9** (по умолч.) | **36.5%** | **56.6%** | **65.3%** | **0.473** |
| 12 | 37.2% | 56.7% | 65.6% | 0.478 |

Учёт контекста поднимает top-1 с **23.7% до 36.5%** (+13 п.п., +54%). На практике:

```
красивая девушк →  девушка, ...      красивые девушк →  девушки, ...   (число)
маленького котён → котёнка, ...       в тёмном лес   →  лесу, ...       (падеж)
детализированный портре → портрет     красивый пейзаж → …
```

**Суффикс-триграмма** (2-словный контекст окончаний) добавлена по итогам ресёрча
и проверена на held-out: она ловит падежное управление предлогом через
прилагательное. Прирост от триграммы:

| | top-1 (все) | top-1 (предложные фразы) |
|---|---:|---:|
| без триграммы | 31.6% | 35.0% |
| **с триграммой** | **34.5%** | **38.8%** |

Словные триграммы обычно разрежены в русском, но здесь они по **окончаниям**
(суффиксов ~6k), поэтому вычислимы и полезны (подход «stem+suffix n-gram»).

**Лексическая словесная биграмма** `P(слово | пред. слово)` добавлена как сигнал
(4): окончания ловят согласование, но не «какое именно слово» уместно (`на
изображении`, `чёткий фокус`). Обучается **только на реальном тексте** (Tatoeba),
целыми словами, с прунингом и капом успешников — остаётся n-граммой (O(1)
lookup, +1.5 МБ, latency без изменений). Свип веса `word_w` на held-out:

| word_w | top-1 | top-3 | top-5 | MRR |
|---:|---:|---:|---:|---:|
| 0 (выкл) | 36.5% | 56.3% | 64.7% | 0.471 |
| 1.2 | 39.6% | 58.2% | 65.9% | 0.495 |
| **2.0** (по умолч.) | **40.2%** | **58.5%** | **65.9%** | **0.499** |
| 3.0 | 40.6% | 58.5% | 65.9% | 0.501 |

Коллокация поднимает top-1 с **36.5% до 40.2%** (+3.7 п.п.), насыщается к ~2.0.
Она же разводит число: `красивые девушк → девушки`, `красивая девушк → девушка`.

### Грамматическое согласование по грамемам (морфология)

Поверхностные n-граммы не ловят согласование надёжно: окончания синкретичны
(`-и` = мн.им. ИЛИ ед.род.) и модель не знает, **какое слово управляет** формой.
Классический не-нейронный ответ — **factored LM** (Bilmes & Kirchhoff): считать
не по форме, а по **грамемам** (POS/род/число/падеж). Чтобы не терять скорость,
морфология вынесена в offline: `src/morph.py` тегирует словарь `pymorphy3` →
`models/morph.tsv.gz`; `src/agreement.py` учит на реальном тексте две крошечные
таблицы — `P(число глагола | число подлежащего)` и `P(род·число прил/местоим |
род·число сущ)`. Рантайм — только словарные лукапы (stdlib, **+0.03 мс**).
Управляющее слово выбирается эвристикой (ближайшее сущ. в им.п. для глагола /
ближайшее сущ. для прил./местоим.).

`python -m src.eval_agreement` — свип веса `AGR_W` на **held-out** кейсах, где
согласование применимо (глагол/прилагательное с найденным управляющим словом):

| AGR_W | top-1 | top-3 | top-5 | MRR |
|---:|---:|---:|---:|---:|
| 0 (выкл) | 26.0% | 47.8% | 59.5% | 0.380 |
| 2.0 | 32.4% | 55.7% | 66.5% | 0.451 |
| **4.0** (по умолч.) | **33.1%** | **57.2%** | **66.9%** | **0.460** |
| 6.0 | 33.6% | 56.3% | 65.5% | 0.456 |

Согласование поднимает top-1 с **26.0% до 33.1%** (+7.1 п.п., +27%) на своих
кейсах, не затрагивая остальные (центрированный скор). На практике:

```
ежик[и] сто…  → стоят/стояли  (мн.)     ежик сто… → стоит  (ед.)
женщина котор… → которую (ж.р.)          мужчина котор… → который (м.р.)
река котор… → которая                    дети игра… → играют
```

Ограничение: «подлежащее» без парсера — эвристика «ближайшее сущ. в им.п.»;
ловит частые порядки `N V`, `Adj N V`, но не инверсию/длинные разрывы.

Ограничение: когда конкурирующее существительное стоит **внутри** причастного/
деепричастного оборота, выделенного запятыми (`девушка, идущая по мосту, красив…`),
надёжно выбрать главу можно только синтаксическим парсером — это вне рамок лёгкой
статистической модели. Обороты через окно обрабатываются, но не идеально.

### Триггер-порог inline-подсказки (когда молчать)

`complete()` всегда возвращает top-k, но серую **inline-подсказку** (`ghost()`)
показываем только при **уверенности** — иначе на коротком неоднозначном огрызке
модель уверенно покажет неверное слово (`из` → `извините`). Порог: длина
префикса ≥3 **и** доминирование top-1 над top-2 по blended-скору ≥ `GHOST_MARGIN`.
Идея из Gmail Smart Compose (arXiv:1906.00080): отдельный триггер решает,
показывать ли предложение вообще. Порог выбран **по данным** —
`python -m src.eval_ghost` строит precision–coverage кривую по-кейстроково на
held-out тексте:

| θ (`GHOST_MARGIN`) | coverage | precision | «полезных»/нажатие |
|---:|---:|---:|---:|
| 0 (показывать всегда) | 99.6% | 53.6% | 53.4% |
| 1.1 | 68.1% | 67.0% | 45.6% |
| **1.4** (по умолч.) | **61.9%** | **70.0%** | 43.4% |
| 3.0 | 39.5% | 83.0% | 32.8% |

Без гейта **≈половина** показанных подсказок неверна (precision 53.6%). Порог
1.4 даёт 70% точных при 62% покрытия (precision — по точному совпадению слова;
на коротком префиксе часть «промахов» — валидные другие словоформы, так что
реальная полезность выше). Тот же порог зашит в браузерный `ghost.html`.

## Запуск

```bash
pip install -r requirements-dev.txt

# 1. подготовка данных и обучение (ngram.json, vocab.tsv, context.json,
#    morph.tsv.gz + agreement.json — последние требуют pymorphy3, build-time)
python -m src.preprocess
python -m src.train
python -m src.export_web        # компактная модель для браузера (models/web_model.json)

# 2. бенчмарки
python -m src.evaluate         # точность дополнения окончаний
python -m src.eval_context     # эффект согласования по окончаниям (свип gamma)
python -m src.eval_agreement   # эффект грамматического согласования (свип AGR_W)
python -m src.eval_ghost       # precision–coverage кривая триггер-порога

# 3. тесты
python tests/test_completer.py        # или: pytest -q

# 4. демо-интерфейс (Streamlit): одно поле с ghost-подсказкой, TAB — принять
streamlit run app.py

# 5. React-фронт (чипы, орбита угла, полукруглая шкала)
cd frontend-react
npm install
npm run dev                           # http://localhost:5173
```

Обученные артефакты уже лежат в `models/`, шаги 1–2 нужны только для
переобучения. Для React-демо нужен `models/web_model.json` (шаг `export_web`);
скрипт `npm run dev` сам подставит его в `public/`. Опциональный LLM next-word:
`python3 scripts/serve.py` (нужен `OPENROUTER_API_KEY`) — Vite проксирует `/next`
на `:8000`. Подробнее: [`frontend-react/README.md`](frontend-react/README.md).

## Встраивание как фича

Весь публичный интерфейс — класс `Completer` (пример: `examples/integration_example.py`):

```python
from src.completer import Completer

completer = Completer.load()                 # один раз при старте сервиса
for s in completer.complete("красивый пейза", k=5):
    s.word     # 'пейзаж'   — полное слово
    s.ending   # 'ж'        — что дописать после набранного
    s.score    # ранжирующий скор
    s.source   # 'vocab' | 'model' | 'vocab+model'
```

Свойства для интеграции: потокобезопасное чтение, без глобального состояния и
сети, детерминированность, `complete()` ~0.5 мс. Для уверенной inline-подсказки —
`completer.ghost("красивый пейза")` (возвращает `Suggestion` или `None`).
Переобучение под домен — подменить `data/ru_full.txt` (или дополнить его
промпт-корпусом) и `python -m src.train`.

## Интерфейс (Streamlit)

`app.py` — одно поле ввода с **inline ghost-подсказкой**: по мере набора серым
показывается наиболее вероятное окончание текущего слова, **TAB** — принять.

Подсказка показывается только при **уверенности** (префикс ≥3 символов и явное
доминирование top-1 — см. «Триггер-порог» выше), иначе поле остаётся пустым.
Символьная n-грамма — порядка **6** (по символам).

Чтобы подсказка появлялась **мгновенно, без задержек**, дополнение в демо
выполняется **целиком в браузере**: `python -m src.export_web` печёт компактную
модель (`models/web_model.json` ≈ 12 МБ: топ-словарь + модели согласования,
коллокаций и **компактные грамемы** для морфологии), которую `frontend/ghost.html`
(чистый JS) грузит один раз и делает поиск локально — **~0.005 мс на нажатие**,
без обращений к серверу. Работают все сигналы, включая **грамматическое
согласование** (`воины сто… → стоят`, `женщина котор… → которую`): грамемы
(4-символьные коды) и крошечные таблицы согласования портированы в JS. Полный
Python-`Completer` (с n-граммой и морфологией всего словаря) остаётся API для
интеграции; страница — его быстрый лёгкий фронт.

> В браузер едут грамемы только для топ-80k форм, поэтому очень редкие
> управляющие слова (напр. `ежики`, ранг ~233k) в демо согласование не включают —
> в Python-API, где размечен весь словарь, работают и они.

> Ранее была версия с round-trip в Python на каждую клавишу — в Streamlit это
> ощутимо тормозит, поэтому логику дополнения вынесли в браузер.

## Дальнейшие шаги (если понадобится точнее)

- **Реальный корпус русских промптов** вместо синтетического — уточнит и частоты
  (сейчас доменный prior — интерполяция из шаблонов + курируемой лексики), и
  лексические коллокации (`на изображении`, `трендовый на artstation`).
- **Синтаксический парсер** (natasha/slovnet) на этапе сборки — точнее находить
  подлежащее/антецедент для согласования на длинных дистанциях и при инверсии.
- **LSTM/квантованный трансформер** как альтернативный бэкенд под тем же
  `Completer` API, если появятся кейсы с длинными зависимостями.
- **Персонализация триггер-порога** — подстраивать `GHOST_MARGIN` под accept-rate
  конкретного пользователя (Smart Compose так и делает).

## Источники

- [hermitdave/FrequencyWords] — частотные списки словоформ (OpenSubtitles 2018).
- [Tatoeba] — открытый набор русских предложений (для модели согласования).
- Romanov & Khusainova, *Evaluation of Morphological Embeddings for the Russian
  Language*, arXiv:2103.06628.
- Обзор open-source корпусов для русского — [natasha/corus], tatianashavrina datasets.
- [pymorphy2/pymorphy3] — морфоанализатор (OpenCorpora) для build-time тегирования.
- Bilmes & Kirchhoff, *Factored Language Models and Generalized Parallel Backoff*
  (HLT-NAACL 2003) — грамемные факторы в n-грамме для флективных языков.
- Chen et al., *Gmail Smart Compose*, arXiv:1906.00080 — триггер-порог показа.
- Патенты по query-autocompletion (subword encoder + n-gram LM + beam search),
  напр. US 11,625,436; US 9,886,432 (stem + suffix n-gram для флективных языков).
